Minions design plan

File support for TaskChat.tsx

Let users attach local files to a task chat turn, upload them into the Minions workspace, and pass reliable file paths to the Hermes agent while preserving streaming chat and projected history.

Scope: chat attachments Primary file: client/src/components/TaskChat.tsx Compared with Hermes WebUI Reviewed: May 10, 2026

Recommendation

Recommended architecture

Use a staged attachment flow: upload files first to a task-scoped workspace directory, then send the chat message as JSON with attachment metadata. The server builds an agent-facing message that includes the uploaded file paths, while the UI renders clean attachment chips instead of exposing transport metadata.

This keeps the current SSE streaming model intact and avoids introducing a Minions-owned message table. Hermes can remain the source of truth for transcript history. Hermes WebUI uses the same broad shape: queue files in the composer, upload before chat start, send attachment metadata with the chat request, and keep attachment metadata out of provider history.

Current Shape

Client

  • TaskChat.tsx owns a single input string and calls sendMessage(taskId, text, settings).
  • useChat.ts posts JSON to POST /api/tasks/:id/messages and subscribes to /live over SSE.
  • FileBrowserPage.tsx already demonstrates multiple file upload through uploadFileEntries().

Server

  • server/routes/chat.ts accepts only content: string for messages.
  • server/routes/files.ts has a Multer-backed upload route, but it requires an existing target directory.
  • server/live-chat.ts stores an in-memory user message and assistant stream snapshot.
  • server/workers/hermes_worker.py projects Hermes SessionDB rows back into TaskMessage objects.

Hermes WebUI Comparison

Patterns worth borrowing

  • static/ui.js keeps S.pendingFiles, renders a tray, dedupes by filename, and uploads files before the chat request.
  • static/messages.js creates a display message with attachments separate from the agent-facing message text.
  • api/routes.py stores pending_user_message and pending_attachments before streaming starts, which improves reload/crash recovery.
  • api/streaming.py strips display-only metadata like attachments before sending history back to providers.
  • Image files get local thumbnail previews in the composer with URL.createObjectURL(), and blob URLs are revoked on remove.
  • Workspace file drag and drop inserts an @path reference instead of re-uploading a file that is already in the workspace.
  • The upload route rejects oversized payloads early with a 20 MB cap and has regression tests for text, binary, too-large, no-file, and bad-session cases.

Patterns not to copy directly

  • Hermes WebUI hand-parses multipart because it uses Python's standard HTTP server. Minions already uses Express and Multer, so keep Multer.
  • Hermes WebUI stores only attachment filenames for display. Minions should keep structured metadata with absolute path, displayPath, size, and mimeType.
  • Hermes WebUI writes uploads into the session workspace root. Minions should use a task-scoped attachment directory to avoid name collisions and workspace clutter.
  • Hermes WebUI appends [Attached files: ...] into message text. Minions should prefer a parseable sentinel block so projected history can remove metadata cleanly.
  • Hermes WebUI has api/file/raw for persisted image previews. Minions currently only has /api/files/download, so image history thumbnails need a raw/preview endpoint or should stay as file badges for MVP.

Data Flow

1. Select User picks files in TaskChat.tsx. Client validates count, size, and duplicates.
2. Upload Client posts files to POST /api/tasks/:id/attachments as multipart/form-data.
3. Store Server writes files under a task-scoped workspace directory and returns metadata.
4. Send Client posts content, settings, and attachments to the current message route.
5. Stream Server starts the existing live run and passes an agent-facing message containing file paths.

Client API helpers

export function uploadTaskAttachments(taskId: string, files: File[]) {
  const formData = new FormData();
  for (const file of files) {
    formData.append('files', file, file.name);
  }

  return request<TaskAttachmentUploadResponse>(
    `/tasks/${encodeURIComponent(taskId)}/attachments`,
    { method: 'POST', body: formData },
  );
}

Server Design

New upload route

Add POST /api/tasks/:id/attachments. It should verify the task exists, create a directory under the Minions workspace, save files there, and return TaskAttachment metadata.

Recommended storage path: ~/.minions/workspace/tasks/<taskId>/attachments/<timestamp-runId>/<filename>.

Message route changes

Extend POST /api/tasks/:id/messages to accept optional attachments. The visible user message remains clean, but the content passed to Hermes includes a structured attachment manifest. 

const displayContent = normalizeUserContent(req.body.content, attachments);
const agentContent = buildAgentContent(displayContent, attachments);

const run = startRun(task.id, sessionId, displayContent, attachments);
void consumeChatRun(runTask, sessionId, agentContent, run.runId);

History and provider safety

Mirror Hermes WebUI's separation between display metadata and provider-safe history. attachments can exist on TaskMessage and LiveChatMessage, but the worker should pass only provider-safe message fields into model history.

Minions already relies on Hermes SessionDB projection, so the cleanest adaptation is: persist the agent-facing sentinel block in Hermes, parse it during projection, return structured attachment metadata to the client, and keep raw attachment JSON out of the visible chat UI.

Agent-facing content

Use a small sentinel block so the worker can later remove it from displayed history while preserving it in the Hermes transcript for agent continuity.

Client Design

TaskChat.tsx

  • Import Paperclip, X, FileText, and optionally AlertCircle from lucide-react.
  • Add a hidden <input type="file" multiple> and a visible icon button in the composer toolbar.
  • Add local draft state: attachments, isUploading, attachmentError, and drag-over state.
  • Extract a local AttachmentTray component if the JSX gets dense.
  • Use Hermes WebUI's thumbnail pattern for images: URL.createObjectURL(file) in draft state and URL.revokeObjectURL() on remove/reset.
  • Use Hermes WebUI's paste pattern for screenshots: intercept image clipboard items, convert each blob to a File, and add it to the same tray.
  • Change submit guard from !text to !text && attachments.length === 0.

useChat.ts

  • Extend sendMessage to accept optional attachments.
  • Include attachments in the JSON body for POST /messages.
  • Keep SSE event handling unchanged except for copying message.attachments in snapshot helpers.
  • Preserve attachment metadata in optimistic and live snapshots so reload/reconnect behavior matches text turns.
  • Update appendLocalSendError so a failed attachments turn still shows the attempted user content.

Submit sequence

async function handleSubmit() {
  const text = input.trim();
  if ((!text && attachments.length === 0) || isStreaming || configPending || isUploading) return;

  setIsUploading(true);
  try {
    const uploaded = attachments.length
      ? await uploadTaskAttachments(taskId, attachments.map(item => item.file))
      : { attachments: [] };

    setInput('');
    setAttachments([]);
    await sendMessage(taskId, text || 'Please use the attached files.', { model, reasoningEffort }, uploaded.attachments);
  } finally {
    setIsUploading(false);
  }
}

Validation And Limits

Area Recommendation Reason
Files per turn Default to 10 Keeps the composer usable and avoids large multipart requests.
File size Start at 20 MB per file, then raise only if needed Matches Hermes WebUI's proven cap and avoids large multipart requests; the current Minions upload route has no explicit Multer limits.
Paths Server-generated task-scoped paths only Prevents a client from attaching arbitrary local paths to a task.
Filename safety Strip separators, normalize names, and avoid collisions Prevents traversal and accidental overwrite.
Binary files Allow upload, but pass paths rather than inlining content Agent tools can decide what can be read; the chat route should not inflate context.
Progress Show indeterminate "Uploading" for MVP fetch does not provide portable upload progress; use XHR only if percentage progress is required.
Image preview Draft thumbnails now, persisted thumbnails later Draft thumbnails can use object URLs immediately; persisted history thumbnails need a Minions raw file route.

Implementation Plan

  1. Types: Add TaskAttachment, TaskAttachmentUploadResponse, and optional attachments on TaskMessage / LiveChatMessage.
  2. Upload endpoint: Add POST /api/tasks/:id/attachments with task lookup, Multer limits, task-scoped path generation, and response metadata.
  3. API helper: Add uploadTaskAttachments() in client/src/lib/api.ts.
  4. Composer UI: Update TaskChat.tsx with file picker, attachment tray, image draft thumbnails, validation messages, drag and drop, and send-state changes.
  5. Send path: Extend useChat.sendMessage(), server/routes/chat.ts, and server/live-chat.ts to carry attachment metadata.
  6. Agent prompt: Build an agent-facing attachment manifest with local paths and keep the visible chat content clean.
  7. Projection: Parse the manifest in hermes_worker.py so refreshed history renders attachment chips.
  8. Optional workspace references: Later, allow dragging existing workspace files into chat as @path references without uploading duplicates.
  9. QA: Run npm run build, test text-only sends, file-only sends, mixed sends, image drafts, rejected files, reload after completion, and concurrent send conflict handling.

Risks And Decisions

Open decisions

  • Should chat support folders on day one, or only multiple files?
  • Is Hermes WebUI's 20 MB per-file limit acceptable for Minions, or do task workflows need a larger cap?
  • Should old attachment directories be cleaned automatically, or retained as task artifacts?
  • Should uploaded files also appear immediately in the existing file browser?
  • Do we want a raw file preview endpoint for persisted image thumbnails, or are file badges enough for the MVP?

Primary risks

  • Transcript noise: avoid raw manifests in chat by updating projection.
  • Path spoofing: verify message attachments were returned by the task upload endpoint and live under the task workspace.
  • Large uploads: add server limits before exposing the composer picker.
  • Orphaned uploads: uploaded files can remain if chat send fails after upload; use task-scoped directories and consider cleanup.
  • Agent capability: uploaded binary files may require existing agent tools or follow-up extraction work.

MVP Acceptance Criteria

  • User can attach one or more files from TaskChat.tsx, remove them before sending, and send with or without text.
  • Files are saved under a task-specific workspace path and never in the app source tree.
  • The assistant receives local file paths in the user turn and can reference them in tool calls.
  • Live streaming behavior remains unchanged after send starts.
  • Reloaded chat history shows the user text and attachment chips, not internal JSON.
  • Text-only chat still behaves exactly as it does today.
